Decisiones de diseño

Architecture Decision Records (ADRs)

Por qué se eligió cada componente clave — alternativas evaluadas y trade-offs aceptados

ADR-01 Dos realms Keycloak (web-realm · mobile-realm) en lugar de uno Activo
Rationale
Audience claims aislados (aud: web-api vs aud: mobile-api) permiten que un token robado de un canal no pueda usarse en el otro. Rate limits y TTLs configurables por canal sin afectar al otro.
Alternativa descartada
Un solo realm con roles diferenciadores. Descartada porque el aud es un claim estándar validable; un rol es solo un claim custom, más fácil de manipular en escenarios de JWT confusion attacks.
Trade-off aceptado
Doble mantenimiento de configuración de realm. Mitigado porque la configuración es declarativa (exportable JSON). En producción: realm-per-product-family sería el siguiente paso.
ADR-02 JWE (RSA-OAEP-256 + A256GCM) sobre las respuestas, independiente de TLS Activo
Rationale
TLS termina en Kong. Entre Kong y los servicios internos, el tráfico viajaría en claro sin JWE. Además, el cifrado E2E protege contra proxies intermedios incluso si se usara TLS mutual hasta el servicio.
Alternativa descartada
TLS extremo-a-extremo (cliente ↔ servicio sin terminar en Kong). Descartada porque Kong necesita inspeccionar el request para aplicar rate-limiting, correlation-id y routing por canal.
Trade-off aceptado
+4KB overhead por respuesta (JWE compact). Lat. adicional ~1ms para RSA-OAEP decrypt en cliente. Aceptable para operaciones financieras donde el payload es pequeño (<2KB).
ADR-03 Pubkey binding stateful (Map en memoria) con TTL = JWT exp Limitado a 1 nodo
Rationale
El binding vincula al usuario (sub) con su clave pública RSA durante la vida del JWT. Usar exp del JWT como TTL elimina la necesidad de cleanup manual y sigue el mismo ciclo de vida que el token.
Alternativa descartada
Incluir la clave pública directamente en el JWT como claim. Descartada porque inflaría el tamaño del token (~800B extra) y requeriría re-autenticación para cambiar la clave.
Trade-off aceptado
El Map vive en el proceso Node. Dos instancias del mismo servicio no comparten el store → un request con pubkey puede registrar en instancia A y fallar en instancia B. Requiere Redis para escalar horizontalmente.
ADR-04 Rate-limiting por token JWT en Kong (no por IP) Activo
Rationale
Rate-limit por IP falla en móvil (IP dinámica, NAT compartido). Limitar por token JWT asegura que el límite sigue al usuario real, no al carrier. Web: 60/min · Mobile: 30/min por diferencia de patrones de uso.
Alternativa descartada
Rate-limit en los propios servicios Node. Descartada porque Kong aplica el límite antes de que el request llegue al servicio, ahorrando CPU de validación JWT en requests que se van a rechazar de todas formas.
Trade-off aceptado
Kong OSS usa store en memoria para los contadores (no Redis). Reinicio de Kong resetea los contadores. Mitigación: Kong rara vez se reinicia; para producción, Kong Gateway + Redis plugin.
Diseño de middlewares

Cadena de middlewares — orden y rationale

El orden no es arbitrario: cada paso falla-rápido antes de ejecutar el siguiente (fail-fast composition)

Cadena — los 3 servicios (GET y POST)
① Kong (gateway)
mTLS · rate-limit por token · correlation-id · channel routing
TLS fail → 400 Rate exceeded → 429
② validateJWT
RS256 · issuer · audience · exp · clockTolerance:30s
Carga JWKS de Keycloak (cacheado en memoria)
Invalid/expired → 401 Wrong audience → 401
③ validateClientInOdoo
POST a Odoo con sub + email + client_id · Header X-Internal-Secret
Verifica active=true en x_api_clientes
Not found / inactive → 403 Odoo down → 502
④ validatePubKeyBinding
Map<sub, {key, exp}> · primer uso registra · reuso verifica
TTL = JWT exp · sin X-Client-Public-Key → skip
Key mismatch → 403
⑤ checkIdempotency (POST únicamente)
X-Idempotency-Key UUID requerido · Map con TTL 5 min
Replay: devuelve respuesta cacheada sin procesar
Sin key → 400 Replay → 200 + Replayed:true
⑥ Handler → respuesta JWE
Lógica de negocio · cifra con pubkey del cliente
Cache-Control: no-store · Content-Type: application/jose
Por qué este orden importa
JWT antes de Odoo

Una llamada a Odoo cuesta ~5ms de I/O. Si el JWT es inválido (caso más frecuente en ataques), rechazar en paso ② evita la llamada a Odoo — CPU y I/O ahorrados.

Odoo antes de Pubkey Binding

Un usuario dado de baja en Odoo no debe poder registrar una clave pública. Si la validación de Odoo fuera después, un atacante podría registrar una clave con un token robado antes de que se detecte la baja.

Idempotency solo en POST

GET es idempotente por semántica HTTP. El middleware se registra solo en rutas POST; no hay overhead en lecturas. El store de idempotencia solo crece con operaciones de escritura.

Riesgo: Odoo como SPOF

Si Odoo está caído, los 3 servicios devuelven 502 en cada request — incluyendo GETs de solo lectura. Pendiente: circuit breaker con fallback configurable (strict / permissive mode).

🚧 Bloqueante para HA

pubKeyStore + idempotencyStore en memoria — cualquier arquitectura active-active requiere Redis. Sin él, los controles de seguridad se vuelven inconsistentes bajo balanceo de carga.

📐 Arquitectura Redis recomendada

Redis Cluster con 2 shards por store. Claves: pkbind:{sub} y idem:{key}. TTL absolutas en UNIX epoch. Evitar KEYS * — usar SCAN para operaciones de mantenimiento.

📊 Estimación de carga en Redis

~100B por entrada de pubkey · ~2KB por entrada de idempotency. 10k usuarios activos simultáneos ≈ 21MB RAM. Trivial para Redis; el cuello de botella es el throughput de Odoo, no Redis.

Patrones implementados

3 patrones de seguridad — diseño y flujo

① Pubkey Binding

Patrón: TOFU (Trust On First Use) con TTL vinculado al ciclo de vida del JWT
// Flujo de decisión
if (!header['x-client-public-key'])
  return next()          // sin binding

const entry = store.get(sub)
const expired = now > entry?.exp

if (!entry || expired) {
  store.set(sub, { key, exp }) // TOFU
  return next()
}
if (entry.key !== key) → 403  // mismatch
next()                           // válido
Propiedades garantizadas:
✓ Un usuario solo puede descifrar con su propia clave
✓ Al renovar JWT, puede cambiar de clave (nuevo TTL)
✓ Sin header → no registra binding (permite readiness probes)

② Anti-Replay Idempotency

Patrón: Idempotency Key con response caching — RFC 9110 §9.2.2 inspirado
// Primera vez
store.set(key, { statusCode, body })
setTimeout(() => store.delete(key), TTL)

// Reenvío (replay)
if (store.has(key)) {
  res.set('X-Idempotency-Replayed', 'true')
  return res.status(cached.statusCode)
            .send(cached.body)  // sin procesar
}

// Wiring: interceptar res.send()
res.send = function(body) {
  if (2xx) store.set(key, ...)
  return orig(body)
}
Propiedades garantizadas:
✓ Mismo UUID → misma respuesta, sin doble proceso
✓ Solo cachea respuestas 2xx (errores no se cachean)
✓ TTL 5 min > TTL JWT 5 min → cobertura total

③ Defense in Depth

Patrón: capas independientes de control — comprometer una no compromete las demás
Capa 1 — Red: mTLS + TLS 1.2/1.3 AEAD ciphers
Capa 2 — Auth: JWT RS256 · audience · issuer · exp
Capa 3 — AuthZ: Odoo active check en los 3 servicios
Capa 4 — Binding: Pubkey TOFU · anti-replay idempotency
Capa 5 — Payload: JWE E2E · no-store · sin metadatos
Un atacante necesita comprometer todas las capas simultáneamente para obtener datos en claro.
