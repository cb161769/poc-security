Estado actual / Objetivo

Odoo hoy vs Keycloak target

El POC valida la migración de web/mobile a Keycloak como broker de identidades, mientras Odoo permanece como motor de autorización de negocio y directorio de clientes.

Estado actual
  • Odoo gestiona usuarios, contraseñas y dispositivos, etc.
  • Keycloak hoy solo federado con Azure AD para herramientas internas.
  • Kong expone las mismas rutas para web y mobile.
  • El directorio de negocio sigue siendo el repositorio de verdad.
Objetivo POC
  • Web/mobile autenticados por Keycloak + PKCE/OIDC.
  • Odoo retiene la autorización de negocio y validación de cliente.
  • Kong separa rutas y audiencias por canal.
  • Se reduce el blast radius de contraseñas en Odoo.

Transición Odoo → Keycloak · Consolidación de realms

Formalizar el plan de migración del acceso web/mobile a Keycloak, mientras Odoo conserva la autorización de negocio y la validación de cliente activo.

⚠ Problema actual — realms separados
El POC usa web-realm y mobile-realm como bases de datos de usuarios independientes.
Un cambio de contraseña en mobile no se refleja en web y viceversa — el usuario existe dos veces sin vínculo entre ellos.
✓ Solución recomendada — realm único con clientes por canal
Consolidar en un solo app-realm. La separación de canales se mantiene por cliente (web-app-client / mobile-app-client), no por realm.
Cada cliente conserva su propio TTL, su propio flujo de autenticación y su audiencia de token. El usuario existe una sola vez — cambia contraseña una vez.
Kong sigue validando la audiencia (aud / azp) por ruta para prevenir cross-channel token reuse.
🛡 Riesgos y mitigaciones
Sesiones no aisladas por canal — un logout web no cierra la sesión mobile.
→ Mitigación: backchannel logout habilitado en ambos clientes + TTL corto en mobile (15 min) hace que la sesión expire sola.
Cross-channel token reuse — un token web podría intentar usarse en endpoints mobile.
→ Mitigación: ya resuelto — Kong valida azp por ruta. Token de web-app-client rechazado en ruta mobile con 401 (verificado en test 08).
Complejidad de migración — usuarios duplicados Odoo ↔ Keycloak durante el corte.
→ Mitigación: ejecutar autenticación vieja y nueva en paralelo durante un sprint de transición. Mapear user_id de Odoo como atributo externalId en Keycloak para mantener trazabilidad.
Passkeys vinculadas al realm — las passkeys registradas en web-realm no se migran automáticamente.
→ Mitigación: re-registro de passkeys post-migración vía Keycloak Account Console (flujo ya implementado con el botón "Registrar Passkey").

Gestión de secretos (Vault)

Las credenciales de base de datos, el usuario admin de Keycloak y las variables de entorno sensibles están hardcodeadas en docker-compose.yml. En producción deben vivir en un gestor de secretos con rotación automática y auditoría de acceso.

DB: Docker Secrets o Vault dynamic credentials
admin-cli: Vault AppRole con TTL corto — token nunca persiste en disco
Keycloak admin: contraseña fuerte + MFA + acceso solo desde red interna

HSM / KMS para llaves RSA

Las llaves RSA que cifran los payloads JWE viven como archivos PEM planos en /shared-keys. Un atacante con acceso al filesystem puede extraer el material de clave.

Cloud: AWS KMS / GCP Cloud KMS — operaciones de firma sin exponer la clave
On-prem: HSM PKCS#11 (Thales / AWS CloudHSM)
Mínimo viable: Android Keystore / iOS Secure Enclave para la clave del cliente
📋 Prioridad media — hardening de infraestructura

Play Integrity API / App Attest

El claim app_version en el JWT verifica versión, pero no integridad del binario. Un APK recompilado o un emulador rootado puede obtener tokens legítimos de Keycloak.

Android: Play Integrity API — atestigua que el APK es el original de Play Store, en hardware real, no rooteado
iOS: App Attest (gratuito) — misma garantía con Secure Enclave
⚠ Play Integrity: gratis hasta 10k req/día; App Attest: gratuito sin límite

SSL Pinning con certificado real

El POC usa un certificado autofirmado — el pin SHA-256 de producción aún no se puede configurar sin un dominio real. En producción se fija el hash del certificado en network_security_config.xml (Android) y Info.plist (iOS).

Almacenamiento seguro del refresh token

El refresh token para biometría se guarda en @capacitor/preferences que usa SharedPreferences plano en Android. En producción debe usar EncryptedSharedPreferences respaldado por Android Keystore, e iOS Keychain.

Segregar rutas Web / Mobile

Kong actualmente expone las mismas rutas para los canales web y mobile. En producción, separar rutas, audiencias y permisos reduce el riesgo de cross-channel token reuse y mejora la trazabilidad de seguridad.

Kong ya en uso Web/Mobile comparten endpoints

Certificados TLS reales

El POC usa certificados autofirmados. En producción se requiere una CA reconocida (Let's Encrypt o CA corporativa) para que los clientes puedan confiar en el servidor sin configuración adicional.

Red interna aislada

Los servicios backend son accesibles desde el exterior en el POC. En producción la red interna debe estar completamente aislada, con Kong como único punto de entrada.

Alertas y monitorización (SIEM)

Los rechazos de seguridad (clave incorrecta, cliente bloqueado, rate limiting, lockdown activado) deben generar alertas. Actualmente solo quedan en logs locales sin visibilidad centralizada.

X-App-Version — Control de UX, no control de seguridad

El header X-App-Version es enviado por el cliente. Como cualquier consultor senior o hacker ético identificaría de inmediato: es trivialmente falsificable.

⚠ El ataque — 1 línea
Cualquier atacante con una versión antigua y vulnerable de la app simplemente envía:
curl -H "X-App-Version: 1.0.0" \
     -H "Authorization: Bearer <token>" \
     https://api.example.com/...
Resultado: el check de versión pasa. Cero seguridad garantizada.
La versión la reporta el cliente. El servidor no puede verificar que sea verdadera.
📋 Qué SÍ hace — y qué NO hace
✓ SÍ Fuerza a usuarios legítimos a actualizar (UX gate)
✓ SÍ Útil para deprecar versiones antiguas en rollout
✗ NO Previene ataques con versiones vulnerables
✗ NO Es un control de seguridad — no pertenece al threat model
✗ NO Detiene herramientas automatizadas (siempre forjan el header)
Clasificación correcta: Upgrade enforcement gate para usuarios honestos. La seguridad real la proveen el JWT RS256 + Odoo AuthZ — no el header de versión.
✓ Fix corto plazo
Documentar explícitamente que X-App-Version no es un trust boundary.
Mover la lógica de versión al claim del JWT emitido por Keycloak al autenticarse — el cliente no puede modificar un claim firmado.
Keycloak puede añadir "app_version": "1.0.0" vía script mapper solo si el client_id corresponde a una versión whitelisted.
🔒 Fix largo plazo — Mobile
Google Play Integrity API (Android)
Google firma una attestation que certifica:
· APK descargado de Play Store (no sideloaded)
· Dispositivo genuino (no emulado)
· APK no modificado/reempaquetado
· Hash exacto del binario instalado
Apple App Attest — equivalente para iOS
→ El atacante no puede forjar esto sin comprometer Google/Apple
🌐 Fix largo plazo — Web
Web no tiene TEE (trusted execution environment) — no existe equivalente a Play Integrity.
Opciones reales:
· Versión en JWT (Keycloak mapper) — unforgeable post-auth
· SRI + CSP estricto — impide modificación del bundle en tránsito
· Short-lived tokens (5 min) — limita la ventana de explotación con versión vieja
En web, asumir el cliente como no confiable es la postura correcta. La defensa está en el servidor.

Pubkey Binding — Sustitución de clave JWE

La primera clave pública del usuario queda registrada; cualquier cambio posterior es rechazado

Vector de ataque

Atacante genera su propio par de claves RSA e inyecta su clave pública en la cabecera X-Client-Public-Key. El servidor cifra la respuesta con la clave del atacante → datos expuestos.

Control implementado

Primera clave del usuario registrada con TTL igual a la validez del JWT. Cualquier request con clave diferente para el mismo usuario devuelve 403. Al expirar el JWT, el binding se libera automáticamente.

// Mapa en memoria: sub → {clave, expiración}
const pubKeyStore = new Map()

function validatePubKeyBinding(req, res, next) {
  const pub = req.headers['x-client-public-key']
  if (!pub) return next()

  const entry = pubKeyStore.get(req.jwt.sub)
  const now = Math.floor(Date.now() / 1000)

  // Primer uso o JWT expirado → registrar
  if (!entry || now > entry.exp) {
    pubKeyStore.set(req.jwt.sub,
      { key: pub, exp: req.jwt.exp })
    return next()
  }
  // Clave diferente → rechazar
  if (entry.key !== pub)
    return res.status(403).json(
      { error: 'Clave de cliente no coincide' })
  next()
}

Anti-Replay — Clave de idempotencia

Prevención de reenvío de operaciones POST en transfers y payments

Vector de ataque

Atacante captura un POST de transferencia válido y lo reenvía varias veces mientras el JWT sigue activo. Sin protección, cada envío genera una nueva transacción.

Control implementado

Cada POST requiere una clave única UUID. Reenvío dentro de 5 minutos devuelve la respuesta original con cabecera X-Idempotency-Replayed: true sin procesar la operación de nuevo.

Respuesta en caso de reenvío
HTTP/1.1 200 OK
X-Idempotency-Replayed: true

[misma respuesta cifrada anterior]
const TTL = 5 * 60 * 1000  // 5 minutos
const store = new Map()

function checkIdempotency(req, res, next) {
  const key = req.headers['x-idempotency-key']
  if (!key)
    return res.status(400).json(
      { error: 'X-Idempotency-Key requerido' })

  // Reenvío → devolver respuesta guardada
  const cached = store.get(key)
  if (cached)
    return res.status(cached.statusCode)
      .set('X-Idempotency-Replayed', 'true')
      .send(cached.body)

  // Primera vez → guardar respuesta
  const orig = res.send.bind(res)
  res.send = function(body) {
    if (res.statusCode >= 200 &&
        res.statusCode < 300)
      store.set(key, {
        statusCode: res.statusCode, body })
    setTimeout(() => store.delete(key), TTL)
    return orig(body)
  }
  next()
}

Passkeys / WebAuthn — Implementación técnica en el POC

Tres vulnerabilidades OIDC encontradas y corregidas durante el desarrollo. El flujo resultante cumple con PKCE + CSRF + WebAuthn Passwordless.

Vuln 1 · ROPC bypass de WebAuthn
Problema: Si directAccessGrantsEnabled=true en web-app-client sin control adicional, cualquiera con credenciales obtiene un token web sin pasar por el flujo WebAuthn.
Diseño actual: ROPC habilitado solo para el formulario inline de la propia app (UX controlado). Creado web-test-client separado con ROPC para la suite de tests. browserFlow=passkey-browser en el realm — el botón passkey usa Auth Code y va por WebAuthn. ROPC y Auth Code son rutas paralelas con propósitos distintos.
✓ Segregado por vía
Vuln 2 · PKCE downgrade
Problema: Sin pkce.code.challenge.method=S256 en los atributos del cliente, un atacante podía iniciar un flujo sin code_challenge y el servidor lo aceptaría, permitiendo intercambiar el auth code si lo intercepta.
Fix: Atributo S256 obligatorio en web-app-client → Keycloak rechaza flujos sin code_challenge.
✓ Corregido
Vuln 3 · CSRF sin state
Problema: El parámetro state en el redirect OIDC no se generaba ni verificaba. Un atacante podía construir un URL de callback con un auth code propio y engañar al usuario para que lo ejecutara (CSRF login attack).
Fix: state generado criptográficamente en sessionStorage y verificado en auth-callback.page.ts.
✓ Corregido
