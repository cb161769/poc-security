Roadmap hacia producción

Los controles actuales cubren los riesgos más críticos. Estos ítems deben resolverse antes de ir a producción

⚠ Prioridad alta — antes de producción

Circuit Breaker

Si Odoo o Keycloak fallan, actualmente todos los endpoints fallan con ellos. Necesitamos un mecanismo que degrade gracefully y proteja la disponibilidad del servicio.

Transición Odoo → Keycloak

Formalizar el plan de migración del acceso web/mobile a Keycloak, mientras Odoo conserva la autorización de negocio y la validación de cliente activo.

Revocación de tokens en tiempo real

Un token comprometido hoy es válido hasta que expira (5 min). Necesitamos poder invalidarlo inmediatamente desde el servidor (logout forzado, cuenta bloqueada).

Stores distribuidos (Redis)

El registro de claves y el control anti-replay viven en memoria del proceso. Un reinicio o escalado horizontal los pierde. Redis garantiza persistencia y consistencia entre nodos.

📋 Prioridad media — hardening de infraestructura

Segregar rutas Web / Mobile

Kong actualmente expone las mismas rutas para los canales web y mobile. En producción, separar rutas, audiencias y permisos reduce el riesgo de cross-channel token reuse y mejora la trazabilidad de seguridad.

Kong ya en uso Web/Mobile comparten endpoints

Certificados TLS reales

El POC usa certificados autofirmados. En producción se requiere una CA reconocida (Let's Encrypt o CA corporativa) para que los clientes puedan confiar en el servidor sin configuración adicional.

Red interna aislada

Los servicios backend son accesibles desde el exterior en el POC. En producción la red interna debe estar completamente aislada, con Kong como único punto de entrada.

Alertas y monitorización (SIEM)

Los rechazos de seguridad (clave incorrecta, cliente bloqueado, rate limiting) deben generar alertas. Actualmente solo quedan en logs locales sin visibilidad centralizada.

Pubkey Binding — Sustitución de clave JWE

La primera clave pública del usuario queda registrada; cualquier cambio posterior es rechazado

Vector de ataque

Atacante genera su propio par de claves RSA e inyecta su clave pública en la cabecera X-Client-Public-Key. El servidor cifra la respuesta con la clave del atacante → datos expuestos.

Control implementado

Primera clave del usuario registrada con TTL igual a la validez del JWT. Cualquier request con clave diferente para el mismo usuario devuelve 403. Al expirar el JWT, el binding se libera automáticamente.

// Mapa en memoria: sub → {clave, expiración}
const pubKeyStore = new Map()

function validatePubKeyBinding(req, res, next) {
  const pub = req.headers['x-client-public-key']
  if (!pub) return next()

  const entry = pubKeyStore.get(req.jwt.sub)
  const now = Math.floor(Date.now() / 1000)

  // Primer uso o JWT expirado → registrar
  if (!entry || now > entry.exp) {
    pubKeyStore.set(req.jwt.sub,
      { key: pub, exp: req.jwt.exp })
    return next()
  }
  // Clave diferente → rechazar
  if (entry.key !== pub)
    return res.status(403).json(
      { error: 'Clave de cliente no coincide' })
  next()
}

Anti-Replay — Clave de idempotencia

Prevención de reenvío de operaciones POST en transfers y payments

Vector de ataque

Atacante captura un POST de transferencia válido y lo reenvía varias veces mientras el JWT sigue activo. Sin protección, cada envío genera una nueva transacción.

Control implementado

Cada POST requiere una clave única UUID. Reenvío dentro de 5 minutos devuelve la respuesta original con cabecera X-Idempotency-Replayed: true sin procesar la operación de nuevo.

Respuesta en caso de reenvío
HTTP/1.1 200 OK
X-Idempotency-Replayed: true

[misma respuesta cifrada anterior]
const TTL = 5 * 60 * 1000  // 5 minutos
const store = new Map()

function checkIdempotency(req, res, next) {
  const key = req.headers['x-idempotency-key']
  if (!key)
    return res.status(400).json(
      { error: 'X-Idempotency-Key requerido' })

  // Reenvío → devolver respuesta guardada
  const cached = store.get(key)
  if (cached)
    return res.status(cached.statusCode)
      .set('X-Idempotency-Replayed', 'true')
      .send(cached.body)

  // Primera vez → guardar respuesta
  const orig = res.send.bind(res)
  res.send = function(body) {
    if (res.statusCode >= 200 &&
        res.statusCode < 300)
      store.set(key, {
        statusCode: res.statusCode, body })
    setTimeout(() => store.delete(key), TTL)
    return orig(body)
  }
  next()
}

Odoo AuthZ — Autorización de negocio

Validación contra el ERP antes de procesar cualquier operación — los 3 servicios

Vector de ataque

async function validateClientInOdoo(req, res, next) {
  try {
    const r = await axios.post(
      `${ODOO_URL}/api/validate-jwt-client`,
      { sub: req.jwt.sub, email: req.jwt.email,
        client_id: req.jwt.client_id,
        roles: req.jwt.roles },
      { headers: {
          'X-Internal-Secret': process.env.INTERNAL_SECRET
        } }
    )
    if (!r.data?.authorized)
      return res.status(403).json(
        { error: 'Cliente no autorizado' })
    req.client = r.data.client
    next()
  } catch (e) {
    return res.status(502).json(
      { error: 'Error verificando autorización' })
  }
}

Análisis estático del APK · Ataques a UI y captura · Checks de build hardening · Red y criptografía

1 Static APK Analysis Descomprime el APK y escanea el binario buscando: URLs http:// (excepto localhost), flag allowBackup, flag debuggable=true, y descriptores DEX cortos (La;, Lb;) como señal de que R8 está obfuscando. MASVS-CODE
MASVS-NETWORK-1		 High ✓
2 FLAG_SECURE Attack Lanza la app y ejecuta adb screencap. Si la captura pesa ≥ 40 KB hay contenido real visible (FLAG_SECURE no está activo). Si pesa 0 o es un frame negro, FLAG_SECURE está bloqueando. Confirma con dumpsys window buscando el bit 0x2000 en los flags de ventana. MASVS-PLATFORM-1 High ✓
3 Root Detection Comprueba que la propiedad ro.build.tags no sea test-keys (firmas de ingeniería). Si lo es, intenta lanzar la app y verifica si la app se cierra sola (detección en runtime). Busca además en el DEX los paths de su binarios (/system/xbin/su, etc.) que la app debe verificar. MASVS-RESILIENCE-1 Medium ✓
4 Frida Detection Busca en el DEX los strings de artefactos Frida: frida-server, re.frida.server, frida-agent. Si están compilados, la app sabe qué buscar en disco. Si Frida está disponible en el contenedor, intenta hacer attach real y verifica si la app lo bloquea. MASVS-RESILIENCE-2 Medium ✓
5 APK Signature (OS) Usa pm path para verificar que el OS reconoció e instaló el paquete correctamente. Si no existe ningún path, el APK fue rechazado por firma inválida o el package name no coincide con el applicationId del build. MASVS-CODE-1 Low ✓
6 WebView CDP Exploit Busca el socket webview_devtools_remote_<PID> en /proc/net/unix. Si existe, hace adb forward tcp:9221 localabstract:webview_devtools_remote_<PID> y envía GET /json al endpoint CDP. Si devuelve webSocketDebuggerUrl, el JS de la app es inspeccionable remotamente. MASVS-PLATFORM-2 Critical ✓
7 Log Leak Captura adb logcat --pid=<PID> filtrando líneas de frameworks del sistema (ActivityManager, WebView, Capacitor Bridge). Si quedan entradas originadas en el código de la app, ProGuard no ha eliminado las llamadas a Log.d/i/w/e. MASVS-CODE-1 Medium ✓
8 Network Security Config Descomprime res/xml/network_security_config.xml del APK y verifica que exista y contenga la directiva cleartextTrafficPermitted. Sin NSC, Android permite HTTP sin TLS en API < 28. MASVS-NETWORK-1 High ⚠
9 JDWP Debugger Attach Ejecuta adb jdwp que lista todos los PIDs de procesos que aceptan conexiones de debugger Java. Si el PID de la app aparece en esa lista, cualquier persona con ADB puede adjuntar jdb y leer/modificar el estado de la JVM en tiempo real. MASVS-RESILIENCE-3 Critical ✓
10 Exported Components Parsea pm dump <pkg> buscando líneas del patrón com.pkg/ClassName: seguidas de exported=true. Excluye el MainActivity launcher. Un componente exportado sin permiso es invocable por cualquier app del dispositivo. MASVS-PLATFORM-1 High ✓
11 ADB Backup Verifica que los flags de pm dump no incluyan ALLOW_BACKUP. Si ese flag está activo, adb backup -apk com.pkg puede extraer todos los datos de la app, incluyendo SharedPreferences y bases de datos SQLite, sin root. MASVS-STORAGE-1 High ✓
12 Tapjacking Busca filterTouchesWhenObscured en los XMLs de layout descomprimidos del APK y en el DEX. Este atributo hace que el sistema operativo ignore toques que llegan a través de una overlay maliciosa (otra app superpuesta de forma transparente). MASVS-PLATFORM-1 Medium ✓
13 Hardcoded Secrets Scan Escanea el binario completo del APK, el DEX y los assets (capacitor.config.json, main.js, vendor.js) con 9 patrones regex: Google API keys (AIza…), AWS access keys (AKIA…), JWT completos, Firebase URLs, Stripe live keys, PEM private keys, OAuth secrets, contraseñas hardcodeadas y API keys. MASVS-STORAGE-1 Critical ✓
14 SSL Certificate Pinning Busca señales de pinning en el DEX: CertificatePinner (OkHttp), TrustKit, X509TrustManager, HostnameVerifier. Verifica network_security_config.xml por directiva <pin-set>. Detecta también onReceivedSslError que podría aceptar certs inválidos con handler.proceed(). MASVS-NETWORK-2 High ⚠
15 WebView Hardening Escanea el DEX buscando configuraciones peligrosas de WebView: setAllowUniversalAccessFromFileURLs (habilita UXSS via file://), MIXED_CONTENT_ALWAYS_ALLOW (HTTP dentro de HTTPS), y addJavascriptInterface (bridge JS → Java). El runtime de Capacitor establece algunos de estos flags; se reporta para auditoría. MASVS-PLATFORM-2 High ⚠

Android Security Tests — Fases 16 a 30

Firma APK · Storage seguro · Ataques IPC · Detección de hooking · Memoria en runtime · Anti-tampering

#FaseQué verifica y cómo lo haceMASVSSev.Run
16 APK Signature Schemes Detecta qué esquemas de firma están presentes: V1 busca META-INF/*.RSA en el ZIP; V2/V3 busca la magic string APK Sig Block 42 en el binario; V4 verifica via apksigner --verbose si está disponible. Solo V1 es vulnerable a ataques de manipulación ZIP. MASVS-CODE-1 High ✓
17 SharedPreferences Audit Con adb root, lista /data/data/<pkg>/shared_prefs/*.xml y parsea el contenido XML buscando atributos name con palabras sensibles (jwt, token, password, pin, oauth, session…). Si encuentra datos en texto plano que deberían estar en EncryptedSharedPreferences, reporta crítico. MASVS-STORAGE-1,2 Critical ✓
18 SQLite Database Audit Con adb root, lista /data/data/<pkg>/databases/*.db y ejecuta sqlite3 .schema sobre cada base. Busca nombres de columna sensibles y verifica si la BD está cifrada (SQLCipher): si sqlite3 falla con error de encabezado, la BD usa cifrado. MASVS-STORAGE-1,2 Critical ✓
19 Deep Link Attack Extrae esquemas URL y autoridades de pm dump. Para cada deep link construido, usa am start -a android.intent.action.VIEW -d <url> con payloads: path traversal (/../../../etc/passwd), SQL injection, XSS y open redirect. Verifica si la app se lanza con el payload. MASVS-PLATFORM-1 Medium ✓
20 Broadcast Injection Identifica BroadcastReceivers exportados en pm dump. Para cada uno ejecuta am broadcast -n <pkg>/<Receiver> sin credenciales. Si el receiver responde al broadcast sin requerir permiso, cualquier app del dispositivo puede activarlo. MASVS-PLATFORM-1 High ✓
21 Content Provider Audit Extrae las autoridades URI de los ContentProviders declarados. Ejecuta adb shell content query --uri content://<authority>/ sin autenticación. Si devuelve filas de datos, el provider está exponiendo información sin requerir permiso (lectura no autorizada de la BD interna). MASVS-PLATFORM-1 Critical ✓
22 Hook Detection (Frida/Xposed) Busca strings de frameworks de hooking en el DEX: XposedBridge, de.robv.android.xposed, LSPosed, Zygisk, Riru, objection. En runtime, verifica /proc/<PID>/maps buscando librerías de Frida inyectadas. Si Frida está en el contenedor, intenta attach real para confirmar si la app lo detecta y termina. MASVS-RESILIENCE-2 Critical ✓
23 Native Library Audit Lista todos los .so del APK. Extrae cada uno y aplica strings (o tr -dc printable). Busca credenciales hardcodeadas con los mismos patrones del Phase 13, y registra si tienen RegisterNatives o JNI_OnLoad (bridge JNI activo). Los .so son frecuentemente ignorados en auditorías. MASVS-CODE-1
MASVS-STORAGE-1		 Medium ✓
24 Memory Scan (runtime) Con adb root, lee el mapa de memoria del proceso (/proc/<PID>/maps) e identifica regiones legibles del heap. Usa dd if=/proc/<PID>/mem para extraer hasta 64 KB de las primeras regiones y busca JWT (eyJ…), claves RSA, credenciales o passwords en texto plano en memoria activa. MASVS-STORAGE-2 Critical ✓
25 Screen Recording Attack Con la app en primer plano, ejecuta adb shell screenrecord --time-limit 2. Si el video resultante pesa > 50 KB hay contenido real grabado (FLAG_SECURE no protege contra screenrecord). Un video pequeño (≤ 10 KB) indica frames negros — FLAG_SECURE activo. Complementa el Phase 2 que solo prueba screencap. MASVS-PLATFORM-1 High ⚠
26 Certificate Transparency Verifica si el NSC incluye requireCertificateTransparency="true" (obliga a que los certs del servidor estén en logs CT públicos). Detecta si el NSC confía en CAs de usuario, lo que permitiría MITM con un certificado instalado manualmente. Busca también la librería okhttp-certificatetransparency en el DEX. MASVS-NETWORK-1 Medium ⚠
27 Clipboard Leak Busca en el DEX uso de ClipboardManager o setPrimaryClip. Verifica si hay patrones de protección: TYPE_TEXT_VARIATION_PASSWORD o setLongClickable(false) en campos sensibles. Campos de contraseña sin inputType=textPassword copian al portapapeles y otras apps pueden leerlo. MASVS-PLATFORM-1 Medium ⚠
28 Emulator Detection Lee las propiedades del sistema: ro.kernel.qemu=1, ro.hardware=goldfish/ranchu, ro.product.model con "sdk" o "emulator", ro.build.fingerprint con "generic". Verifica si el DEX contiene código que compruebe estas propiedades para bloquear ejecución en emuladores/entornos de análisis. MASVS-RESILIENCE-1 Medium ⚠
29 Anti-Tamper / Integrity Busca en el DEX señales de que la app verifica su propia integridad: llamadas a PackageManager.getPackageInfo para comparar firma, cálculos de checksum (CRC, SHA-256), uso de la API SafetyNet / Play Integrity. Sin estos controles, un APK reempaquetado con código malicioso no es detectado. MASVS-RESILIENCE-2 High ✓
30 Intent Injection Para cada componente exportado, lanza 3 intentos de am start: sin extras, con extras que simulan bypass de auth (--es authenticated true --es role admin), y con URI de datos maliciosa (-d file:///etc/passwd). Si la app lanza algún flujo sensible sin validar los extras, hay escalada de privilegios por Intent. MASVS-PLATFORM-1 High ✓

Passkeys / WebAuthn — Implementación técnica en el POC

Tres vulnerabilidades OIDC encontradas y corregidas durante el desarrollo. El flujo resultante cumple con PKCE + CSRF + WebAuthn Passwordless.

Vuln 1 · Direct Grant bypass
Problema: directAccessGrantsEnabled=true en web-app-client permitía obtener tokens vía ROPC (usuario+contraseña) sin pasar por WebAuthn, anulando el flujo de passkeys.
Fix: directAccessGrantsEnabled=false en web-app-client. Creado web-test-client separado con ROPC solo para automatización de tests.
✓ Corregido
Vuln 2 · PKCE downgrade
Problema: Sin pkce.code.challenge.method=S256 en los atributos del cliente, un atacante podía iniciar un flujo sin code_challenge y el servidor lo aceptaría, permitiendo intercambiar el auth code si lo intercepta.
Fix: Atributo S256 obligatorio en web-app-client → Keycloak rechaza flujos sin code_challenge.
✓ Corregido
Vuln 3 · CSRF sin state
Problema: El parámetro state en el redirect OIDC no se generaba ni verificaba. Un atacante podía construir un URL de callback con un auth code propio y engañar al usuario para que lo ejecutara (CSRF login attack).
Fix: state generado criptográficamente en sessionStorage y verificado en auth-callback.page.ts.
✓ Corregido
