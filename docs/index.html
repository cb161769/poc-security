Controles P1a

Pubkey Binding — JWE Key Substitution

Prevención de sustitución de clave pública en respuestas JWE

⚔️ Vector de ataque

Atacante intercepta la request e inyecta su propia clave pública en X-Client-Public-Key. El servidor cifra la respuesta con la clave del atacante → datos expuestos.

🛡️ Control implementado

Primera clave enviada por un sub queda registrada. Cualquier request posterior con clave diferente → 403. TTL = JWT exp para cleanup automático.

// Map — persiste en memoria
const pubKeyStore = new Map()

function validatePubKeyBinding(req, res, next) {
  const pub = req.headers['x-client-public-key']
  if (!pub) return next()   // header opcional

  const entry = pubKeyStore.get(req.jwt.sub)
  const now = Math.floor(Date.now() / 1000)

  // Primer uso o TTL expirado → registrar
  if (!entry || now > entry.exp) {
    pubKeyStore.set(req.jwt.sub,
      { key: pub, exp: req.jwt.exp })
    return next()
  }
  // Clave diferente → bloquear
  if (entry.key !== pub)
    return res.status(403).json(
      { error: 'Clave no coincide' })
  next()
}
Controles P1b

Anti-Replay — Idempotency Key

Prevención de reenvío de operaciones POST (transfers + payments)

⚔️ Vector de ataque

Atacante captura un POST de transferencia válido y lo reenvía múltiples veces con el mismo JWT. Sin protección, cada reenvío ejecuta una nueva transacción.

🛡️ Control implementado

X-Idempotency-Key (UUID) requerido en todo POST. Reenvío dentro de 5 min → respuesta cacheada + header X-Idempotency-Replayed: true. Sin duplicados.

Headers en replay
HTTP/1.1 200 OK
X-Idempotency-Replayed: true
Content-Type: application/jose

[misma respuesta cacheada]
const IDEMPOTENCY_TTL_MS = 5 * 60 * 1000
const store = new Map()

function checkIdempotency(req, res, next) {
  const key = req.headers['x-idempotency-key']
  if (!key)
    return res.status(400).json(
      { error: 'X-Idempotency-Key requerido' })

  // Replay detectado → respuesta cacheada
  const entry = store.get(key)
  if (entry)
    return res.status(entry.statusCode)
      .set('X-Idempotency-Replayed', 'true')
      .send(entry.body)

  // Primera vez → interceptar send para cachear
  const orig = res.send.bind(res)
  res.send = function(body) {
    if (res.statusCode >= 200 && res.statusCode < 300)
      store.set(key, {
        statusCode: res.statusCode,
        contentType: res.getHeader('Content-Type'),
        body
      })
    setTimeout(() => store.delete(key), IDEMPOTENCY_TTL_MS)
    return orig(body)
  }
  next()
}
Controles P1c

Odoo AuthZ — Autorización de Negocio

Validación de clientes activos contra ERP antes de procesar cualquier request

⚔️ Ataque

JWT válido pero cliente cancelado/suspendido en el ERP. Sin AuthZ de negocio, cualquier usuario con token puede operar.

🛡️ Control

Cada request valida sub del JWT contra modelo x_api_clientes en Odoo. active=false → 403 inmediato.

Modelo Odoo

email   = Char()

# active_test:False para buscar
// validateClientInOdoo middleware
async function validateClientInOdoo(req, res, next) {
  try {
    const r = await axios.post(
      `${ODOO_URL}/api/validate-jwt-client`,
      { sub: req.jwt.sub, email: req.jwt.email,
        client_id: req.jwt.client_id,
        roles: req.jwt.roles },
      { headers: { 'X-Internal-Secret': SECRET } }
    )
    if (!r.data?.authorized)
      return res.status(403).json(
        { error: 'Cliente no autorizado por Odoo' })
    req.client = r.data.client
    next()
  } catch (e) {
    return res.status(502).json(
      { error: 'Error validando cliente' })
  }
}

Roadmap hacia Producción

Brechas identificadas para hardening adicional en entorno productivo

⚠️ PRIORIDAD ALTA

P2 — Circuit Breaker

Resilience4j / Opossum para proteger servicios backend cuando Odoo o Keycloak están degradados. Sin él, una caída de Odoo tumba todos los endpoints.

P2 — Refresh Token Rotation

Rotation + revocation en Keycloak. Token comprometido actualmente válido hasta expiración. Blacklist Redis para revocación inmediata.

P2 — Distributed Idempotency

Store en memoria = no persistente entre restarts. Redis con TTL para producción multi-instancia.

📋 PRIORIDAD MEDIA

P3 — Pubkey Store Persistente

Map en memoria se pierde en restart. Persistir en Redis/DB con TTL = JWT exp. Sin esto, restart de servicio invalida todos los bindings activos.

P3 — Alerting y SIEM

Exportar eventos de seguridad (403 pubkey, Odoo 403, rate-limit 429) a sistema de alertas. Actualmente solo logs locales.

P3 — secure-net: internal: true

Docker Compose tiene internal: false por conveniencia del POC. En producción, aislar completamente la red backend del mundo exterior.
