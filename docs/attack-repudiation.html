Identity & Authentication

JWT Replay & Token Theft

Mitigado
💀 Vector de ataque
  • →
    Atacante intercepta un JWT válido en tránsito o desde storage del cliente
  • →
    Reutiliza el token en requests posteriores, haciéndose pasar por el usuario legítimo
  • →
    Si el token no expira o tiene TTL largo, la ventana de ataque es grande
  • →
    Token válido puede ser usado desde cualquier IP, device o canal
🛡️ Controles implementados
  • ✓
    TTL 5 minutos en ambos reinos — ventana de replay mínima
  • ✓
    RS256 + JWKS — firma verificada con clave pública de Keycloak, no se puede forjar
  • ✓
    JWT en memoria del servicio Angular — no expuesto en localStorage/sessionStorage
  • ✓
    clockTolerance: 30s — margen mínimo para drift de relojes sin ampliar ventana
// Keycloak realm config accessTokenLifespan: 300 // 5 min // jsonwebtoken verify { algorithms: ['RS256'], clockTolerance: 30 }
web-realm TTL 5 min
mobile-realm TTL 5 min
JWT en memoria (no storage)
Sin token revocation (roadmap)
02 / 08
Channel Isolation

Cross-Channel Token Attack

Mitigado
💀 Vector de ataque
  • →
    Usuario obtiene token del canal web (menor restricción de rate limit: 60/min)
  • →
    Reutiliza ese token en endpoints del canal mobile para evadir controles diferenciados
  • →
    Sin validación de canal, el mismo JWT permitiría acceso en ambos contextos
  • →
    Permite saltar políticas de seguridad específicas del canal (e.g., mobile: biometría, device binding)
🛡️ Controles implementados
  • ✓
    Reinos separados — web-realm ≠ mobile-realm, JWKS distintos, issuers distintos
  • ✓
    audience claim por canal — web-api vs mobile-api
  • ✓
    Validación cruzada en cada servicio: token web en endpoint mobile → 401
  • ✓
    Rate limits diferenciados — 60/min web · 30/min mobile aplicados por Kong
// Test de cross-channel — resultado esperado curl web-token → /api/v1/mobile/transfers ← 401 { "error": "JWT inválido" } // iss=web-realm rechazado en contexto mobile
2 reinos Keycloak aislados
aud web-api / mobile-api
Verificado en test-all.sh
03 / 08
JWE Encryption

JWE Key Substitution Attack

Mitigado
💀 Vector de ataque
  • →
    Atacante genera su propia clave RSA y envía la clave pública en X-Client-Public-Key
  • →
    El servidor cifra la respuesta JWE con esa clave arbitraria — atacante puede descifrar
  • →
    Usuario legítimo recibe JWE que no puede descifrar con su clave real
  • →
    Permite al atacante leer datos sensibles cifrados destinados al usuario víctima
🛡️ Controles implementados
  • ✓
    Pubkey binding por sub — primera request registra la clave para ese JWT subject
  • ✓
    Clave en memoria del proceso — Map<sub, base64JWK> no es alterable desde fuera
  • ✓
    Cambio de clave → 403 — request con clave diferente para el mismo sub es rechazada
  • ✓
    Endpoint explícito POST /api/v1/data/register-key para flujos mobile con registro previo
// Primera request: registra PUB_KEY_A para sub:123 pubKeyStore.set('sub:123', PUB_KEY_A) ✓ // Segunda request con clave diferente pubKeyStore.get('sub:123') !== PUB_KEY_B ← 403 { "error": "Clave no coincide con la registrada" }

Binding en payments-service

04 / 08
Transaction Integrity

API Replay / Double-Submit

Mitigado
💀 Vector de ataque
  • →
    Atacante captura una request POST válida de transferencia y la reenvía múltiples veces
  • →
    Con JWT aún válido (dentro del TTL de 5 min), el servidor la procesa todas las veces
  • →
    Resultado: débitos múltiples del mismo monto para la misma operación legítima
  • →
    También aplica a errores de red donde el cliente reintenta automáticamente
🛡️ Controles implementados
  • ✓
    X-Idempotency-Key requerido en todo POST de transferencias y pagos
  • ✓
    TTL 5 minutos por clave — Map con setTimeout de limpieza automática
  • ✓
    Respuesta cacheada en replay — misma respuesta JWE, header X-Idempotency-Replayed: true
  • ✓
    Sin clave → 400 inmediato — previene requests no controladas
// POST /transfers con UUID único por operación X-Idempotency-Key: a3f2b1c4-9e8d-... // Segundo intento con mismo UUID → cache hit ← 200 X-Idempotency-Replayed: true // No se procesa de nuevo — respuesta idéntica
POST /transfers protegido
POST /payments protegido
TTL auto-limpieza 5 min
05 / 08
Business Authorization

AuthZ Bypass / Unauthorized Operations

Mitigado
💀 Vector de ataque
  • →
    Usuario con JWT válido intenta acceder a transfers/payments sin estar autorizado en el sistema de negocio
  • →
    JWT solo prueba identidad, no autorización de negocio — son capas distintas
  • →
    Sin validación Odoo en todos los servicios, un token válido es suficiente para operar
  • →
    Cliente desactivado en Odoo podría seguir operando si solo transfers/payments validan JWT
🛡️ Controles implementados
  • ✓
    Odoo AuthZ en los 3 servicios — api-node, transfers y payments consultan x_api_clientes
  • ✓
    X-Internal-Secret — canal seguro entre servicios Node y Odoo
  • ✓
    Validación de monto — transfers ≤$10k, payments ≤$5k, métodos permitidos: card/ach/wire
  • ✓
    Rol JWT + Odoo — doble capa: rol de Keycloak + registro activo en Odoo
POST /api/validate-jwt-client (Odoo) { sub, email, client_id, roles } ← { authorized: true, client: { name, limits } } // Cliente inactivo o no registrado ← 403 { "error": "Cliente no autorizado por Odoo" }

06 / 08
Information Disclosure

Framework & Payload Fingerprinting

Mitigado
💀 Vector de ataque
  • →
    X-Powered-By: Express — revela el framework usado, facilita búsqueda de CVEs específicos de versión
  • →
    ETag en respuestas JWE — el hash del payload revela el tamaño exacto del ciphertext, útil para ataques de oracle de compresión
  • →
    Combinado con timing attacks permite inferir contenido cifrado por variaciones de tamaño
  • →
    Stack traces en errores revelan paths internos, versiones y estructura del código
🛡️ Controles implementados
  • ✓
    app.disable('x-powered-by') en los 3 servicios — header eliminado de todas las respuestas
  • ✓
    app.set('etag', false) — sin ETag en ninguna respuesta, JWE incluido
  • ✓
    Cache-Control: no-store en todas las respuestas JWE — previene caching intermediario
  • ✓
    Kong agrega Via: kong/3.5.0 — controlado y sin información interna de servicios backend
// Antes (brechas confirmadas en auditoría viva) X-Powered-By: Express ← eliminado ETag: W/"1a2b-abc..." ← eliminado Cache-Control: (ausente) ← agregado: no-store
X-Powered-By eliminado
ETag desactivado
Cache-Control: no-store en JWE
07 / 08
Availability & Abuse

Rate Abuse & Token Flooding

Mitigado
💀 Vector de ataque
  • →
    Atacante con JWT válido envía cientos de requests para saturar el backend (JWKS, Odoo, base de datos)
  • →
    Rate limiting por IP es bypassable con múltiples IPs o proxies — el token es el identificador real
  • →
    Sin límite por token, un solo usuario puede generar carga desproporcionada en servicios downstream
  • →
    Abuso de endpoints costosos (Odoo AuthZ) puede generar degradación de servicio para todos los usuarios
🛡️ Controles implementados
  • ✓
    Rate limiting por token JWT — Kong: limit_by: header, header_name: Authorization
  • ✓
    Límites diferenciados por canal — web: 60 req/min · mobile: 30 req/min
  • ✓
    JWKS cache 10 min — limita requests a Keycloak a 10/min por cliente JWKS
  • ✓
    Kong como perímetro único — todo el tráfico pasa por Kong antes de llegar a los servicios
// kong.yml — rate limiting por token plugins: - name: rate-limiting config: minute: 30 # mobile limit_by: header header_name: Authorization policy: local
Por token, no por IP
60/min web · 30/min mobile
Policy: local (1 nodo — roadmap: Redis multi-nodo)
08 / 08
Transport Security

Man-in-the-Middle & Eavesdropping

Parcial
💀 Vector de ataque
  • →
    Atacante en la red intercepta el tráfico entre el cliente y Kong (red WiFi, proxy corporativo, etc.)
  • →
    Sin TLS, puede leer el JWT en claro y el JWE compacto (aunque el payload está cifrado)
  • →
    Con el JWT robado tiene hasta 5 minutos para operar como el usuario
  • →
    TráfIco interno Docker (entre servicios) no pasa por TLS si la red es de confianza solo por configuración
🛡️ Controles implementados / pendientes
  • ✓
    TLS en Kong — certificado autofirmado en POC, reemplazar por Let's Encrypt/CA corporativa en prod
  • ✓
    JWE end-to-end — incluso si TLS es terminado por proxy intermedio, el payload sigue cifrado RSA-OAEP-256 + A256GCM
  • ✓
    HSTS + headers de seguridad configurados en Kong para todos los endpoints
  • ⚠
    Red interna Docker — secure-net: internal: false en POC. Cambiar a internal: true en producción
// docker-compose.yml — pendiente en prod secure-net: internal: true # ← cambiar de false // Kong TLS — reemplazar certs autofirmados ssl_cert: /shared-keys/fullchain.pem ← Let's Encrypt
